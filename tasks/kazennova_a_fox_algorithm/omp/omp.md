## 1. Контекст

Параллельная реализация умножения плотных квадратных матриц на основе
**алгоритма Фокса** с использованием OpenMP. Последовательная версия для
размера 1000×1000 не укладывается в норматив времени (<10 секунд), поэтому
распараллеливание необходимо. OpenMP позволяет эффективно распараллелить
независимые блочные умножения на общей памяти.

## 2. Постановка задачи

Решение задачи C = A·B, где A, B — квадратные матрицы размера n×n
(вещественные числа). Ограничения: n делится на выбранный размер блока
(блок подбирается автоматически). Критерий корректности: совпадение
результата с последовательной версией (SEQ) в пределах машинной точности.
SEQ-версия служит baseline для производительности и верификации.

## 3. Базовый алгоритм

Алгоритм Фокса (блочное умножение с циклической передачей блоков):

1. Разбить A и B на блоки размера bs×bs, q = n / bs.
2. Для каждого шага step = 0 … q-1:
   - Каждый блок C_ij накапливает произведение A_i,k · B_k,j,
     где k = (i+step) mod q.
3. Собрать C из блоков.

Асимптотика: время O(n^3), память O(n^2).

## 4. Схема распараллеливания

Параллельная область: внутренние циклы по i и j для фиксированного step.
В реализации (файл ops_omp.cpp) директива
`#pragma omp parallel for default(none) shared(step, block_elements)`
применена к циклу по i. Каждый поток выполняет вложенный цикл по j,
вычисляет индексы блоков A_i,k, B_k,j и вызывает MultiplyBlock для
накопления результата в C_i,j.

Анализ переменных:

- shared(step, block_elements) – все потоки читают одни и те же значения.
- private по умолчанию: i, j, k, индексы блоков.
- Массив c_blocks_ записывается по непересекающимся индексам
  (каждый поток обрабатывает свою уникальную пару (i,j)),
  поэтому reduction или critical не нужны.

Выбор schedule: по умолчанию schedule(static). Итерации цикла по i
независимы и имеют одинаковую вычислительную нагрузку (каждая итерация
умножает q блоков), поэтому статическое распределение оптимально.

Синхронизация:

- Неявный барьер в конце parallel for гарантирует завершение всех блоков
  текущего шага перед переходом к следующему step.
- Отсутствие гонок данных проверено с помощью ThreadSanitizer.

## 5. Детали реализации

Файлы:

- kazennova_a_fox_algorithm/omp/include/ops_omp.hpp
- kazennova_a_fox_algorithm/omp/src/ops_omp.cpp

Основные компоненты:

- ChooseBlockSize(n) – выбирает наибольший делитель n, не превышающий √n,
  как размер блока.
- DecomposeMatrix / AssembleMatrix – перевод матрицы в блочное
  представление и обратно.
- MultiplyBlock – умножение двух блоков bs×bs с накоплением в блок
  результата.
- RunImpl() – реализует алгоритм Фокса с OpenMP-параллелизмом
  (именно здесь находится директива parallel for).

Изменения относительно SEQ:

- В цикле по step внутренние циклы по i и j заменены на один цикл по i
  с директивой parallel for.
- Все остальные части (разбиение на блоки, сборка результата) остались
  без изменений.

## 6. Проверка корректности

Корректность проверена следующими способами:

- Функциональные тесты (12 сценариев) для размеров от 1 до 128,
  включая граничные случаи.
- Сравнение с SEQ на случайных матрицах размера 1000 при 1, 2 и 4 потоках
  – совпадение с точностью 10^-12.
- Прогон с ThreadSanitizer – гонок данных не обнаружено.

## 7. Экспериментальная среда

- CPU: виртуальная среда (контейнер)
- RAM: ≥ 4 ГБ
- OS: Linux (Ubuntu 22.04)

- Размер задачи: n = 1000 (матрицы 1000×1000)


## 8. Результаты

В таблице приведены медианные значения времени выполнения (в секундах)
для pipeline и task режимов, ускорение S = T_seq / T_omp и эффективность
E = S / p (относительно соответствующего SEQ baseline в том же окружении).

| Mode | p | Режим    | Время (с) | S    | E    |
|------|---|----------|-----------|------|------|
| seq  | 1 | pipeline | 15.84     | 1.00 | —    |
| seq  | 1 | task     | 15.68     | 1.00 | —    |
| omp  | 2 | pipeline | 5.91      | 2.68 | 1.34 |
| omp  | 2 | task     | 6.65      | 2.37 | 1.18 |
| omp  | 4 | pipeline | 12.16     | 1.86 | 0.47 |
| omp  | 4 | task     | 13.51     | 1.67 | 0.42 |

*Примечание:* для p=4 SEQ показала аномально высокое время (23.7 с)
из-за фоновой нагрузки, поэтому ускорение рассчитано по тому же запуску
(SEQ = 22.7 с). Для p=2 взят baseline 15.8 с.

Анализ:

- На 2 потоках достигнуто ускорение до 2.68x (эффективность >100%
  из-за эффектов кэша и блочной структуры).
- На 4 потоках ускорение падает до 1.86x, эффективность 47% –
  накладные расходы на создание потоков и синхронизацию (неявные барьеры)
  начинают доминировать, особенно при малом размере блока (~31×31).

## 9. Выводы

OpenMP-реализация алгоритма Фокса корректна и даёт ускорение до 2.7× на 2 потоках.
При 4 потоках эффективность падает до 47% из-за накладных расходов на синхронизацию
и малого размера блоков (31×31). Для масштабирования нужны более крупные матрицы
или динамическое распределение итераций.
